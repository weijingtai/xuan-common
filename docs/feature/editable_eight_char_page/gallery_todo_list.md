  - 输出：缩略预览 Widget 的 props 契约（必须/可选字段）
  - 验收：用一段明确列表描述 props（例如 template/selected/favorite/onTap 等）

### A2. 缩略预览组件（先做 UI 框架，后做真实预览）
- [ ] A2.1 新增缩略预览 Widget：TemplateThumbnailCard（占位版本）
  - 输入：template、selected、favorite、onTap
  - 操作：实现固定宽高的卡片容器（圆角/边框/背景），内部先用占位（例如灰色骨架块）

  - 验收：Gallery/Album item 已替换为该 widget，且无布局溢出/抖动

  - 输入：当前 Gallery bar 高度与布局
  - 操作：将 bar 高度调整到可展示缩略图（建议 92~104），保持横向贯穿屏幕
  - 输出：更高的 Gallery bar + 横向滚动缩略项
  - 验收：缩略项可辨识，右侧“更多”按钮仍可用且不挤压内容

### A3. 实时渲染真实卡片缩略图（核心）
- [ ] A3.1 选择渲染路线并落地（优先路线 1，失败则切路线 2）
  - 输入：现有卡片渲染组件与所需运行时依赖
  - 操作：
    - 路线 1（优先）：用现有卡片渲染组件生成静态预览（缩放 + clip）
    - 路线 2（兜底）：自绘/简化版预览（只呈现 cardStyle + 行列结构的关键视觉）
  - 输出：TemplateThumbnailCard 内部展示“真实卡片视觉”
  - 验收：不同模板在缩略图上能看出差异（背景/边框/分隔/字体大小层级等至少 2 个维度不同）

- [ ] A3.2 缩略图内容裁剪与缩放策略
  - 输入：目标缩略尺寸（Gallery/Album）
  - 操作：确定 preview 的 logical size，并用 Transform.scale/ClipRRect 保持比例与清晰度
  - 输出：稳定的缩放策略（不会因屏幕尺寸变化导致内容被挤压变形）
  - 验收：多分辨率窗口下缩略图不会溢出、不会出现明显拉伸

- [ ] A3.3 叠加信息（不破坏“看样式”的主目标）
  - 输入：template.name、template.description、favorite/selected 状态
  - 操作：
    - 右上角：favorite 星标（小图标）
    - 选中态：外边框高亮（primary）
    - 悬停（桌面）：浮层显示 name/description（不改变 item 尺寸）
  - 输出：缩略图可视化 + 最少辅助信息
  - 验收：鼠标 hover 不产生 overflow；选中态一眼可见

## 方案 2：Base64 图片缓存（规划 + 可逐步替换）

说明：方案 2 的目标是把“缩略预览”变成图片（Base64/bytes）缓存，Gallery/Album 直接渲染图片以获得最佳滚动性能。实现顺序建议：先实现生成与存储，再接入 UI“优先读缓存、无缓存回退方案 1”。

### B1. 数据模型与存储方案（Base64）
- [ ] B1.1 定义缩略图数据结构
  - 输入：模板 id、缩略图尺寸策略（Gallery/Album）
  - 操作：定义 thumbnail 字段（建议区分 small/medium，或用单一尺寸 + 缩放）
  - 输出：数据结构说明（字段名/格式/base64 编码规范）
  - 验收：明确 base64 编码的 mime（png/webp）与尺寸约束

- [ ] B1.2 存储位置决策与接入点
  - 输入：当前模板持久化方式（templateJson）
  - 操作：选择存储方式：
    - 方式 1：写入 templateJson（字段 thumbnailBase64）
    - 方式 2：独立表/独立 KV（templateId -> base64）
  - 输出：确定一种方式，并列出需要改动的持久化层位置
  - 验收：能明确列出“写入/读取/删除”各发生在何处

---

### B2. 缩略图生成机制（Render -> Image -> Base64）
- [ ] B2.1 实现“从 Widget 生成 png bytes”的工具函数
  - 输入：要渲染的 preview widget、目标尺寸
  - 操作：使用 RepaintBoundary/toImage（或等价方案）生成 image bytes
  - 输出：`Future<Uint8List>` 的生成函数
  - 验收：对同一个模板生成的图片稳定；尺寸符合预期

- [ ] B2.2 Base64 编解码与大小控制
  - 输入：png bytes
  - 操作：base64Encode/base64Decode，并增加尺寸/质量策略（避免过大）
  - 输出：`String` base64（或 data-uri）与还原函数
  - 验收：缓存后重新加载能正常显示；单张体积可控（建议给出上限，例如 < 200KB）

---

### B3. 生成时机与一致性
- [ ] B3.1 在“保存/另存为/复制”后生成缩略图并持久化
  - 输入：保存链路位置（ViewModel 的保存方法）
  - 操作：在保存成功后触发生成（避免阻塞 UI：可先保存模板，再异步补图并二次保存/更新存储）
  - 输出：模板更新后具备 thumbnailBase64
  - 验收：保存后再次进入页面，缩略图无需实时渲染即可展示

- [ ] B3.2 删除/重置的清理策略
  - 输入：delete/reset 逻辑
  - 操作：删除模板时同步删除 thumbnail；重置时清空全部 thumbnail
  - 输出：无孤儿缓存
  - 验收：删除/重置后存储中不存在对应 thumbnail 数据

---

### B4. UI 接入（优先读缓存，回退实时渲染）
- [ ] B4.1 Gallery item：优先使用 base64 图片
  - 输入：thumbnailBase64（可能为空）
  - 操作：如果有缓存，直接 Image.memory；否则回退方案 1 的实时渲染
  - 输出：混合策略 item
  - 验收：有缓存时滚动明显更顺；无缓存也可正常展示

- [ ] B4.2 Album item：同样策略 + 更大尺寸
  - 输入：medium 缩略或 small 放大
  - 操作：与 Gallery 一致
  - 输出：统一策略
  - 验收：Album 首屏加载速度可接受，滚动顺畅

---

### B5. 风险与回滚
- [ ] B5.1 失败兜底与容错
  - 输入：生成失败/解码失败/数据过大
  - 操作：捕获错误并回退实时渲染；必要时清除坏缓存
  - 输出：稳定性策略
  - 验收：任何失败不影响模板选择与编辑

- [ ] B5.2 渐进式上线开关
  - 输入：配置开关（常量/运行时 flag）
  - 操作：提供一个可快速禁用缓存图的开关
  - 输出：快速回滚能力
  - 验收：关闭开关后完全使用方案 1
