# 命理排盘核心架构：定生辰、大运与多维历法拆分解耦方案 (PRD)

**文档版本**: v1.0
**核心议题**: 解决“一源多态”的生辰排盘冲突、人工校正接入问题，以及大运和高阶历法产物的持久化难题。

---

## 1. 背景与当前业务困境：“定生辰”的挑战

在玄学排盘系统中，“定生辰”（从一个输入的公历时间，转化为用于算命的四柱八字）并非一个简单的 $f(t) = result$ 的纯数学映射，而是一个**极其复杂的、受多维玄学流派配置和人为干预影响**的过程。

### 1.1 生成一件“八字”的四大核心变量漏斗

一个单纯的 `DateTime` 输入，在变成排盘所需干支之前，必须经过以下四层干预变量的过滤：

1. **时间标准（Time Standard）**：程序功能必须完整支持以下四种基准：**标准时**、**夏令时**、**移除夏令时**、**平太阳时**、**真太阳时**。时间标准的切换直接扭曲了计算的绝对时间坐标。
2. **出生地经纬度（Coordinates）**：推算真伪太阳时和天体运行的绝对前置条件，且有着极高的精度要求（差之毫厘的坐标导致跨子时或跨节气，排盘彻底改变）。
3. **子时划分策略（Zi-Shi Strategy）**：23点不分早晚子（直接换次日），还是0点划分早晚子且套用五鼠遁？这导致**日柱、时柱**产生巨大分歧。
4. **节气定点与精度（Jie-Qi Specs）**：交节采用平气法还是现代天文定气法？交切精度是按分钟还是只按时辰？这导致**年柱、月柱**产生变异。

### 1.2 人工干预与“反推”的困境

真实业务场景中，除了客观计算外，占卜师往往面临**主观干预**的需求：

1. **时间微调（校正）**：客观真太阳时算出是某时辰，但占卜师根据命主的事实反馈，反推其出生处于“刻分界限”，手动将时间前拨或后仰15分钟。
2. **纯手工强排（盲派盲填）**：求测人只有流传下来的干支八字（甚至来自古籍），**完全没有公历时间坐标**。

**核心困境结论**：一个输入源存在**N种正确结果**的“叠加态”，此外还有脱离了现代时间坐标系客观规律的**人工主观干预态**。

## 2. 过去的妥协解决方案及其痛点 (`info_list_json`)

为了应对上述一源多果的情况，过去的系统在 `Seekers` 和 `TimingDivinations` 主表中采用了以下方案：

- 将部分历法属性（`yearGanZhi` 等）直接提平到了主表的列里面。
- 将不同时间标准算出来的结果打包成一个 `List<DivinationDatetimeModel>`，强行序列化为 JSON 字符串，存入名为 `info_list_json`（即 `timingInfoListJson`）的单列中。

### 这种方案的设计缺陷 (Anti-Patterns)

1. **违背第一范式，剥夺 SQL 能力**：几十种历法信息被压缩成了一坨文本。数据库成为了没有检索能力的“存放砖块的文件系统”。（例如：无法用 SQL `WHERE` 查出所有的真太阳时盘）。
2. **性能塌方与数据膨胀**：`DivinationDatetimeModel` 对象极其庞大（含经纬度、各套甲子、节气流）。仅仅因为想展示一个用户列表，就不得不把海量废弃的 JSON 塞进内存。
3. **架构的迁移灾难（Migration Nightmare）**：一旦天文算法库底层或 `DivinationDatetimeModel` 结构变动，当解析旧版 JSON 数据时会引发级联反序列化崩溃（`TypeError`），维护极为痛苦。
4. **无法包容“手动强排”记录**：一个需要经纬度推演的天文历法 JSON 数组里，如果强行塞入一个没有 `DateTime` 基准的手工盲派八字，将直接导致底层时间转换引擎 Panic。

## 3. 全新的解决方案：分层、解耦、完全关系化

为了一劳永逸解决“定生辰”与后续“大运”（乃至胎元等）推演的顽疾，我们采用**独立建表、依赖倒置**的方案。

主表（如 `Seekers`）**彻底剥离**庞大且冗余的复合历法 JSON 数组。

- 仅保留：用户的原始输入（输入的标准公历时间、性别、姓名等）。
- **【核心新增】：活跃定盘指针 (`current_calendar_uuid`)**。主表中新增一个外键字段，严格指向此时此刻卦师最终采信的那一条 `DivinationCalendar` 的 UUID。
- **字段保留**：`yearGanZhi`（生肖年干支）**不应被删除**，它在系统的其他功能（例如基础的生肖速查、大类归档）中仍具备极高的独立使用价值。
- **删除废弃物**：彻底废弃那饱受诟病的 `info_list_json`（即 `timingInfoListJson`）字段。

### 第二层：设立历法快照中心 (`DivinationCalendars`)

建立独立的 `t_divination_calendars` 数据库表。这是对“历法组合环境”的快照留存。

- **关联设计**：通过 `source_uuid`（指向 Seeker 等）实现 1 对 N。一个用户可以拥有多条不同的排盘快照。
- **算法环境快照存储**：将 `timingType`, `ziStrategy`, `jieQiType` 作为独立列，详细记录推算出下面这套干支**使用了什么规则**。
- **硬核能力：包容手动校正与强排**
  - 新增 `is_manual` (Bool) 标识，说明这条干支结果叠加了占卜师的主观干预。
  - 新增 **`manual_description` (Text)** 字段。当卦师进行反推校准时，强制要求填写描述信（如：“真太阳时此时辰命主反馈不符，性格偏向上一时辰，故手动前拨15分钟定为午时”），这在之后的复盘中极具参考价值。
  - **核心设计**：`calculated_datetime` 列设为 **Nullable (`NULL`)**。以此完美兼容纯盲派干支：允许只存四柱干支，没有时间源。
  - **版本控制与回退（Append-Only History）**：在真实的占卜语境中，卦师可能会面临“二次校准”，或者在校准后发现“还是原本废弃的那套真太阳时更准”并希望**切回原盘**。因此，`DivinationCalendars` 本质上是一个**“只追加（Append-Only）”的快照病历本**：
    - 对已“手动校正”的盘进行**二次校准**时，生成一条全新的 `DivinationCalendar`，并把主表的 `current_calendar_uuid` 指针移向新数据。旧的手工盘依然保留。
    - 想要**切回被遗弃的生辰**时，只需在 UI 列表中点击那条原始的“真太阳时（自动计算）”盘，将主表的 `current_calendar_uuid` 重新指向它即可。
    - 系统绝不执行 `DELETE` 或硬面 `UPDATE` 去覆盖任何“被认为错误”的盘面快照，充分保障测算的完整交叉对比体系。

### 第三层：设立高阶推演表 (`DaYunRecords` 大运快照)

大运作为建立在八字干支与节气之上的高阶产物，它的起始年龄受平定气法影响甚至相差一天以上。

- **关联设计**：通过 `calendar_uuid` 强绑定到上述第二层的某一条具体的 `DivinationCalendar` 记录上（或者绑定回源 `sourceUuid` 加上其所用的 `jieQiType` 配置）。
- **计算结果封存**：将算出的大运柱组打包序列化存入本表的 `pillars_json` 中，确保无论引擎如何升级，旧版本被固化不丢失。
- **降级容灾策略（重中之重）**：
  当遭遇第二层中 `calculated_datetime = NULL` 的手工纯干支八字时，由于失去了追溯下一个节气的公历坐标，大运算法失去算力崩溃。此时，系统自动捕捉异常并**降级（Degrade）**：在交互页面弹窗，强制要求占卜师手动录入大运起运年龄（如“3岁交运”），从而生成对应的 `DaYunRecord` 平等存入表中。

## 4. 方案的可行性与风险分析

### 可行性分析

1. **存储层面（SQLite + Drift）**：高度可行。关系型数据库生来就是做 1对N 拆分关联的。使用 UUID 作为关联键非常自然，查询速度将有百倍提升（避免了恶劣的文本反序列化）。
2. **计算层面（Dart 代码架构）**：我们无需改变底层的推算逻辑本身，只需更改“拼装与存储”的时机。原本是塞进一个 List 对象，现在是并行分发插入多行 SQL 记录。
3. **极限可扩展性（以“刻分干支”和“胎元”为例）**：
   - **刻分支持**：拆分后，对于占卜师要求的更细粒度的“刻”、“分”推演，我们只需在 `DivinationCalendars` 增加 `keGanZhi` 和 `fenGanZhi` 作为独立列（同为 `intEnum<JiaZi>()`）。这就完美顺滑地从“四柱”系统拓展为了“六柱”系统，且原有的存储架构和对大运的依赖不需要做任何破坏性的重构。
   - **胎元支持**：胎元（受孕月份的干支）通常是基于月柱倒推10个月（或依据特定太阳到达某黄经角度推算）。拆分后，胎元的生成与大运非常类似。我们既可以直接将 `taiYuanGanZhi` 作为一列加入 `DivinationCalendars` 中（因为它是生辰的孪生兄弟），也可以效仿大运单独起表。由于现在的表结构极度整洁（一层一层往上搭积木），介入任何新维度的推演都不会再引发 JSON 解析器崩溃的蝴蝶效应。

### 潜在风险预测

1. **数据迁移成本高（Migration Tax）**：这是唯一的阵痛。上线的版本需要编写严密的 Dart 数据迁移脚本。读取旧有的、臃肿的 `info_list_json`，用尽力气做最后一次反序列化，拆解成平级对象，循环生成并插入到全新的 `DivinationCalendars` 表中。
2. **过度生成的脏数据（Orphan Data Bloat）**：如果用户频繁在 UI 界面拉动“真太阳时”/“标准时”开关，会导致 `DivinationCalendars` 表快速生成大量只看了一秒钟的废弃快照。
   - **应对策略**：只有用户点击“保存该盘面”或者明确要求持久化比对时，才写入数据库；或者通过定期的清理脚本（判断无 Panel 引用的 Calendar）进行 GC 垃圾回收。

## 5. 用户指南与前端体验升级 (User Guidelines)

这套底层的革命性重构，将直观传导到前端 User 体验的巨大提升：

1. **“一键比对 / 随心切换” 体验**
   用户在排盘界面可以点击顶部的一个 Segment Control 切换“真太阳时 / 平太阳时”。由于所有数据已经分行生成存在独立的表里，切换操作变成了毫秒级的纯视图渲染（仅仅是读取不同的 UUID），再也没有任何计算等待。甚至可以实现“左右分屏视图，左边开真太阳时看早子时，右边开平太阳时看晚子时”。
2. **“强排盲干支” 体验**
   占卜师在创建面板时，可以选择切换到“纯手工录入 Tab”。不选时间，直接拨盘选择四柱干支。底部会友好地弹出一个提示：“检测到无基准时间录入，为确保排盘准确，请手动指定大运起运岁数：[下拉框]”。
3. **“定频追溯” 验证体验**
   用户查阅半年前的一个命局，发现它呈现的是大运完全不一样的状态。通过关联的算法策略字段，排盘表头清晰地显示一个标签：`手工校正+平气法+晚子时版本`。占卜师对半年前做了什么人工干预一目了然。

## 6. 深入解析：胎元、胎月、胎日的计算、变量与存储解耦

“胎元”作为生命起点的干支能量，与大运同等重要。我们基于当前代码基底（`lib/features/tai_yuan/`），彻底梳理它的计算方法及在全解耦新架构下的生息位置。

### 6.1 我们当前提供“胎日”和“胎月”的计算吗？

**结论是：完全支持，且在不同算法下提供的信息厚度不同。**
“胎元”在传统命理学中原本特指**“受孕月份的干支”（即胎月）**。但在我们当前的业务引擎中，根据选用的推算流派，系统能够直接推演出包含**胎年、胎月、胎日、胎时**在内的**受孕全四柱（胎八字）**，自然也就包含了“胎日”和“胎月”。

### 6.2 胎元的四种计算方案与核心变量

系统内建了 4 套计算流派（`TaiYuanCalculateStrategy`），不同的流派所需变量和得出的精度完全不同：

#### 方案 A：月柱推算法 (`monthPillarMethod`) —— 经典盲派法

- **得出的精度**：仅有“胎月”（干支组合 `JiaZi`）。
- **核心变量**：只有命主的【出生月柱】。
- **计算逻辑**：天干向后顺推一位（`+1`），地支向后顺推三位（`+3`）。（如：甲子月生，胎月为乙卯）。
- **特点**：古代最常用的粗算方法，完全不需要具体的公历时间源，哪怕是手工强制排的纯盲盘也能算出。

#### 方案 B：日主阴阳法 (`dayMasterYinYangMethod`)

- **得出的精度**：仅有“胎月”。
- **核心变量**：命主的【出生日主阴阳】、【出生月支】、【出生时支】。
- **计算逻辑**：
  - 阳干日主：取出生【月支】中的藏干为胎月天干；地支直接照搬月支。
  - 阴干日主：取出生【时支】中的藏干为胎月天干；地支照搬月支。
- **特点**：涉及地支藏干理论，属于特定流派。同样不需要具体公历源。

#### 方案 C：生日倒推法 (`birthdayCountbackMethod`) —— 精确定轨法

- **得出的精度**：推算出**受孕当日的完整公历时间和节气**，从而得出完整的**孕期四柱（含胎月、胎日）**。
- **核心变量**：命主的【绝对出生公历时间 (`calculatedDatetime`)】、【实际孕育天数（`actual_mature_days`）】。
  - **流派分歧支持**：传统按“十月怀胎”通常减去 300 天，但现代医学或某些流派主张减去 280 天。此架构支持作为入参传入 `actualMatureDays`，动态生成不同孕期的胎元记录。
- **计算逻辑**：从排盘基准公历时间减去传入的天数（如 300 天或 280 天），得出精准的受孕公历时间，再依据此时间进行标准的天文排盘。
- **局限性**：如果遇到了 `isManual=true` 且没有公历时间源的纯手工盘，此算法**彻底罢工**。

#### 方案 D：确切受孕日直接排盘 (`directConceptionDate`)

- **得出的精度**：完整的孕期四柱（含胎月、胎日）。
- **核心变量**：【人工输入的受孕公历时间（甚至精确到时辰）】。
- **计算逻辑**：跳过任何推算，直接就着输入的公历坐标排生成盘。

### 6.3 胎元的存储方式与 Seekers/生辰 之间的关系

在过去的 `info_list_json` 乱象中，由于胎元模型（`TaiYuanModel`）过于臃肿（包含了策略、修正信息、试管婴儿标记等），根本没有做持久化，只是在 UI 层即时算即时扔。

但在我们全新的三层解耦架构下，胎元的持久化关系可以用一句简单的话概括：
**“胎元是八字（DivinationCalendars）的孪生兄弟，大运（DaYunRecords）的亲叔伯”。**

由于胎元与大运一样，高度依赖“算法策略”这一变量，它**绝不能存储在主表 Seekers 当中**。我们为其设计 **独立的胎元快照表 (`t_tai_yuan_records`)**，它的挂载逻辑如下：

- **谁生出了这个胎元？**
  `t_tai_yuan_records` 表通过 `calendar_uuid` 作为外键，严格从属于 `t_divination_calendars` 中的**某一个具体星历快照**。
  
- **为什么要分离计算态？（完美支持多版本共存与回溯）**
  假如针对同一个真太阳时盘（`calendar_uuid=X`），卦师既想看“传统月柱法”算出的简单胎月，又想看如果“早产早产20天、按绝对日子倒推”算出来的胎日到底是在哪个节气前；或者在 280 天和 300 天的流派之间摇摆。
  这就是为什么我们需要关系型分离：所有被算出的胎元都会**被无损记录下来**。为了实现“无感回溯切换”，真正的纽带桥梁在于：**`DivinationCalendars` 主表中也要增加一个“当前选定的胎元指针 (`current_tai_yuan_uuid`)”**。这样只要在这个字段上换个 ID，界面上的胎元立马就跟着切了，而所有历史尝试过的数据全部作为“归档记录”安全躺在 `TaiYuanRecords` 表里。

- **数据表的下潜归档（具体数据结构）：**
  我们通过审阅代码中的 `TaiYuanModel`，结合关系型数据库查询优势，将核心关联字段“独立成列”，将非结构化的附属对象“打包为 JSON”。最终落库的 `TaiYuanRecords` 表结构如下：

  ```dart
  class TaiYuanRecords extends Table {
    // 1. 本纪录唯一标识与关联外键
    TextColumn get uuid => text().named('uuid')(); 
    TextColumn get calendarUuid => text().named('calendar_uuid')(); 
    
    // 2. 算法与前置变量特征
    IntColumn get strategy => intEnum<TaiYuanCalculateStrategy>(); // 当时用的哪种流派（枚举：倒推/阴阳/月柱等）
    DateTimeColumn get conceptionDateTime => dateTime().nullable()(); // 精确的受孕时间（倒推法/人工输入法特有）
    IntColumn get actualMatureDays => integer().withDefault(const Constant(300)).named('actual_mature_days')(); // 实际推算的孕期天数（精准支持 280天 / 300天 流派）
    RealColumn get totalMatureMonth => real().named('total_mature_month')(); // 月数换算（鉴别早晚产，标准为10）
    BoolColumn get isTestTubeBaby => boolean().withDefault(const Constant(false))(); // 试管婴儿标识
    
    // 3. 核心计算结果枚举值（独立成列，方便 SQL 统计和倒推查询）
    IntColumn get taiYuanGanZhi => intEnum<JiaZi>().named('tai_yuan_gan_zhi')(); // 原生胎元（胎月）
    IntColumn get adjustedTaiYuan => intEnum<JiaZi>().nullable().named('adjusted_tai_yuan')(); // 因早晚产调整后的最终胎元
    IntColumn get taiYuanBeforeMonth => integer().named('tai_yuan_before_month')(); // 胎元与出生月相差的月份
    
    // 4. 高级附属八字模型（完整受孕四柱）
    // 通过生日倒推法(birthdayCountback)算出的完整 DateTime 和 EightChars（含胎日胎时），使用 JSON 整体序列化封存入 text() 列，保障字段扩展性并不受应用层增减字段影响。
    TextColumn get extendPayloadJson => text().nullable().named('extend_payload_json')();
  }
  ```
